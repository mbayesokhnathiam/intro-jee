Introduction générale

Qu'est-ce que Jakarta EE ?

Jakarta EE (Jakarta Enterprise Edition) est une plateforme de développement d'applications d'entreprise basée sur Java. Elle fournit un ensemble complet de spécifications, d'APIs et de services pour créer des applications robustes, sécurisées et scalables.

Objectif principal

Simplifier le développement d'applications d'entreprise en fournissant des APIs standardisées et des services intégrés.

Usage typique

Applications web complexes, services web RESTful, systèmes de gestion d'entreprise, microservices.

Objectifs pédagogiques du cours

Développer des applications pratiques

Comprendre l'utilisation de Jakarta EE en entreprise

  • Concevoir des architectures multi-couches robustes
  • Développer des APIs REST avec JAX-RS

  • Déployer des applications sur des serveurs d'applications

Historique et évolution

Pourquoi Java EE est devenu Jakarta EE ?

Contexte

En 2017, Oracle a transféré Java EE à la Eclipse Foundation pour accélérer l'innovation et permettre une gouvernance plus ouverte.

Problème de nom

Oracle a conservé les droits sur le nom "Java", obligeant Eclipse Foundation à adopter le nom "Jakarta EE" (du nom de la capitale indonésienne).

Chronologie de l'évolution

  • Introduction des annotations
  • EJB 3.0 simplifié
  • JPA introduit
2009

Java EE 6

  • CDI (Contexts and Dependency Injection)
  • JAX-RS pour REST
  • Bean Validation
2013

Java EE 7

  • WebSocket API
  • JSON Processing
  • Batch Applications
2017

Java EE 8

  • JSON-Binding
  • HTTP/2 support
  • Security API
2019

Jakarta EE 8

  • Transfert vers Eclipse Foundation
  • Changement de namespace
  • Nouvelle gouvernance
2020

Jakarta EE 9

  • Migration namespace javax → jakarta
  • Java 8+ requis
  • Modernisation APIs
2022

Jakarta EE 10 Actuelle

  • Java 21 support
  • Nouvelles fonctionnalités
  • Performance améliorée

Avantages de cette architecture

Séparation des responsabilités

Chaque couche a un rôle bien défini, facilitant la maintenance et les tests.

Scalabilité

Possibilité de faire évoluer chaque couche indépendamment selon les besoins.

Réutilisabilité

Les composants métier peuvent être réutilisés par différentes interfaces.

Fonctionnalités clés

CDI - Injection de dépendances

Contexts and Dependency Injection permet de gérer automatiquement les dépendances entre objets et leurs cycles de vie.

Avantages principaux

  • Couplage faible entre composants
  • Gestion automatique du cycle de vie
  • Support des événements applicatifs
  • Intercepteurs et décorateurs

Exemple de code

@ApplicationScoped
public class UserService {
    
    @Inject
    private UserRepository userRepository;
    
    @Inject
    private Logger logger;
    
    public User createUser(User user) {
        logger.info("Creating user: " + user.getEmail());
        return userRepository.save(user);
    }
}

JAX-RS - APIs RESTful

Java API for RESTful Web Services facilite la création d'APIs REST avec des annotations simples.

Avantages principaux

  • Annotations intuitives (@GET, @POST, etc.)
  • Sérialisation JSON automatique
  • Gestion des codes de statut HTTP
  • Support des filtres et intercepteurs

Exemple de code

@Path("/users")
@ApplicationScoped
public class UserResource {
    
    @Inject
    private UserService userService;
    
    @GET
    @Produces(MediaType.APPLICATION_JSON)
    public List<User> getAllUsers() {
        return userService.findAll();
    }
    
    @POST
    @Consumes(MediaType.APPLICATION_JSON)
    public Response createUser(User user) {
        User created = userService.create(user);
        return Response.status(201).entity(created).build();
    }
}

JPA - Persistance des données

Java Persistence API offre un mapping objet-relationnel puissant pour gérer la persistance des données.

Avantages principaux

  • Mapping objet-relationnel automatique
  • Langage de requête JPQL
  • Gestion des relations complexes
  • Cache de second niveau

Exemple de code

    private List<Order> orders = new ArrayList<>();
    
    // Getters et setters...
}

Jakarta Security

Framework de sécurité intégré pour l'authentification, l'autorisation et la protection des applications.

Avantages principaux

  • Authentification déclarative
  • Contrôle d'accès basé sur les rôles
  • Protection CSRF intégrée
  • Support OAuth2 et JWT

Exemple de code

@Path("/secure")
public class SecureResource {
    
    @GET
    @RolesAllowed("ADMIN")
    @Produces(MediaType.APPLICATION_JSON)
    public Response getAdminData() {
        return Response.ok("Données admin").build();
    }
    
    @GET
    @Path("/user")
    @RolesAllowed({"USER", "ADMIN"})
    public Response getUserData() {
        return Response.ok("Données utilisateur").build();
    }
}

Gestion des transactions

JTA (Java Transaction API) permet de gérer les transactions de manière déclarative ou programmatique.

Avantages principaux

  • Transactions déclaratives avec @Transactional
  • Support des transactions distribuées
  • Rollback automatique en cas d'exception
  • Gestion des niveaux d'isolation

Exemple de code

@ApplicationScoped
public class OrderService {
    
    @Inject
    private OrderRepository orderRepository;
    
    @Inject
    private PaymentService paymentService;
    
    @Transactional
    public Order processOrder(Order order) {
        // Sauvegarde de la commande
        Order savedOrder = orderRepository.save(order);
        
        // Traitement du paiement
        paymentService.processPayment(order.getAmount());
        
        // Si une exception survient, tout est annulé
        return savedOrder;
    }
}

Comparaison Jakarta EE vs Spring Boot

Deux approches, même objectif

Jakarta EE et Spring Boot sont deux solutions populaires pour le développement d'applications Java d'entreprise. Chacune a ses avantages selon le contexte du projet.

Comparaison détaillée

Critère Jakarta EE Spring Boot Avantage
Philosophie Spécifications standards, implémentations multiples Framework complet avec écosystème intégré
Configuration Annotations + XML minimal, Convention over Configuration Annotations + Java Config, très flexible
Injection de dépendances CDI natif, puissant et standardisé DI Spring, mature et feature-rich
APIs REST JAX-RS standard, annotations déclaratives Spring MVC + Spring WebFlux, très populaire
Persistance JPA standard avec implémentations multiples Spring Data + JPA, abstraction simplifiée
Sécurité Jakarta Security, intégré mais moins mature Spring Security, très mature et complet
Standards Suit les standards JCP/Jakarta EE Crée ses propres standards de facto
Portabilité Très portable entre serveurs conformes Dépendant de l'écosystème Spring

Points forts de chaque solution

Jakarta EE

  • Standards ouverts et portables
  • Forte adoption en entreprise
  • Serveurs d'applications robustes
  • Écosystème mature et stable
  • Indépendance vis-à-vis des vendors

Spring Boot

  • Écosystème très riche
  • Documentation excellente
  • Communauté très active
  • Développement rapide
  • Support microservices complet

Quand utiliser quelle solution ?

Choisir Jakarta EE quand...

  • Applications d'entreprise complexes
  • Environnements avec serveurs d'applications
  • Besoin de portabilité entre providers
  • Projets nécessitant des standards stricts
  • Équipes familières avec Java EE

Choisir Spring Boot quand...

  • Développement rapide d'applications
  • Projets microservices
  • Startups et équipes agiles
  • Applications cloud-native
  • Besoin d'un écosystème complet

Conseil pour votre choix

Le choix entre Jakarta EE et Spring Boot dépend largement du contexte de votre projet, de votre équipe et de vos contraintes techniques.

Pour débuter : Spring Boot est souvent plus accessible grâce à sa documentation et sa communauté. Pour l'entreprise : Jakarta EE offre plus de standardisation et de portabilité à long terme.

Entité JPA - Book.java

@Entity
@Table(name = "books")
public class Book {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @Column(nullable = false)
    private String title;
    
    @Column(nullable = false)
    private String isbn;
    
    @ManyToOne
    @JoinColumn(name = "author_id")
    private Author author;
    
    @Column(name = "publication_year")
    private Integer publicationYear;
    
    @Enumerated(EnumType.STRING)
    private BookStatus status = BookStatus.AVAILABLE;
    
    // Constructeurs
    public Book() {}
    
    public Book(String title, String isbn, Author author) {
        this.title = title;
        this.isbn = isbn;
        this.author = author;
    }
    
    // Getters et setters...
}

enum BookStatus {
    AVAILABLE, BORROWED, RESERVED
}

Service métier - BookService.java

@ApplicationScoped
public class BookService {
    
    @PersistenceContext
    private EntityManager em;
    
    @Inject
    private Logger logger;
    
    @Transactional
    public Book createBook(Book book) {
        logger.info("Creating book: " + book.getTitle());
        
        // Validation métier
        if (bookExists(book.getIsbn())) {
            throw new IllegalArgumentException("Book with ISBN already exists");
        }
        
        em.persist(book);
        return book;
    }
    
    public List<Book> findAll() {
        return em.createQuery("SELECT b FROM Book b", Book.class)
                 .getResultList();
    }
    
    public List<Book> findByAuthor(String authorName) {
        return em.createQuery(
            "SELECT b FROM Book b WHERE b.author.name LIKE :name", Book.class)
                 .setParameter("name", "%" + authorName + "%")
                 .getResultList();
    }
    
    @Transactional
    public Book borrowBook(Long bookId, Long userId) {
        Book book = em.find(Book.class, bookId);
        
        if (book == null) {
            throw new EntityNotFoundException("Book not found");
        }
        
        if (book.getStatus() != BookStatus.AVAILABLE) {
            throw new IllegalStateException("Book not available");
        }
        
        book.setStatus(BookStatus.BORROWED);
        return book;
    }
    
    private boolean bookExists(String isbn) {
        Long count = em.createQuery(
            "SELECT COUNT(b) FROM Book b WHERE b.isbn = :isbn", Long.class)
                       .setParameter("isbn", isbn)
                       .getSingleResult();
        return count > 0;
    }
}

API REST - BookResource.java

@Path("/books")
@ApplicationScoped
@Produces(MediaType.APPLICATION_JSON)
@Consumes(MediaType.APPLICATION_JSON)
public class BookResource {
    
    @Inject
    private BookService bookService;
    
    @GET
    public Response getAllBooks() {
        List<Book> books = bookService.findAll();
        return Response.ok(books).build();
    }
    
    @GET
    @Path("/{id}")
    public Response getBook(@PathParam("id") Long id) {
        try {
            Book book = bookService.findById(id);
            return Response.ok(book).build();
        } catch (EntityNotFoundException e) {
            return Response.status(404)
                          .entity("Book not found")
                          .build();
        }
    }
    
    @POST
    public Response createBook(@Valid Book book) {
        try {
            Book created = bookService.createBook(book);
            return Response.status(201)
                          .entity(created)
                          .build();
        } catch (IllegalArgumentException e) {
            return Response.status(400)
                          .entity(e.getMessage())
                          .build();
        }
    }
    
    @PUT
    @Path("/{id}/borrow")
    public Response borrowBook(@PathParam("id") Long bookId, 
                              @QueryParam("userId") Long userId) {
        try {
            Book book = bookService.borrowBook(bookId, userId);
            return Response.ok(book).build();
        } catch (IllegalStateException e) {
            return Response.status(409)
                          .entity(e.getMessage())
                          .build();
        }
    }
    
    @GET
    @Path("/search")
    public Response searchByAuthor(@QueryParam("author") String author) {
        List<Book> books = bookService.findByAuthor(author);
        return Response.ok(books).build();
    }
}
